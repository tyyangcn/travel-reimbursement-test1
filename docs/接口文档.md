# 差旅报销管理系统接口文档

## 1. 基本说明

- 服务地址：`http://localhost:8082`
- 接口前缀：`/api`
- 数据格式：`application/json`
- 日期格式：`yyyy-MM-dd`
- 状态字典：`0-草稿`、`1-审批中`、`2-已作废`

统一响应格式：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

## 2. 报销单接口

### 2.1 分页查询报销单

- 请求方式：`GET`
- 请求地址：`/api/reimburse/page`
- 请求参数：

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| reimNo | String | 否 | 报销单号，支持模糊查询 |
| reimTitle | String | 否 | 报销标题，支持模糊查询 |
| travelReason | String | 否 | 出差事由，支持模糊查询 |
| companyId | String | 否 | 费用归属公司 ID |
| reimDeptId | String | 否 | 报销部门 ID |
| reimUserId | String | 否 | 报销人 ID |
| businessTypeId | String | 否 | 业务类型 ID |
| status | Integer | 否 | 单据状态 |
| pageNum | Long | 否 | 页码，默认 1 |
| pageSize | Long | 否 | 每页数量，默认 10 |

示例：`GET /api/reimburse/page?pageNum=1&pageSize=10&status=0`

### 2.2 查询报销单详情

- 请求方式：`GET`
- 请求地址：`/api/reimburse/{id}`
- 路径参数：`id`，报销单主键

### 2.3 新增报销单

- 请求方式：`POST`
- 请求地址：`/api/reimburse`
- 请求体：

```json
{
  "documentDate": "2026-06-30",
  "reimTitle": "上海项目出差报销",
  "travelReason": "前往客户现场进行项目验收",
  "reimUserId": "EMP001",
  "reimUserNo": "202101497",
  "reimUserName": "徐年年",
  "reimDeptId": "DEPT001",
  "reimDeptNo": "CS001",
  "reimDeptName": "测试部",
  "companyId": "COMP001",
  "companyNo": "GS001",
  "companyName": "初始公司",
  "businessTypeId": "BT001",
  "businessTypeNo": "RC01",
  "businessTypeName": "日常办公",
  "remark": "新增接口测试"
}
```

成功时 `data` 返回新报销单的 32 位主键，新增单据初始状态为草稿。

### 2.4 修改报销单

- 请求方式：`PUT`
- 请求地址：`/api/reimburse/{id}`
- 路径参数：`id`，报销单主键
- 请求体：与新增接口相同
- 业务规则：只有草稿状态的报销单允许修改。

### 2.5 送审报销单

- 请求方式：`POST`
- 请求地址：`/api/reimburse/{id}/submit`
- 路径参数：`id`，报销单主键
- 业务规则：草稿送审后状态由 `0` 变为 `1`，即审批中。

送审前后端均校验：至少存在一条行程、行程均已生成补助、分摊比例合计为
100%，且分摊金额合计等于补助总金额。

### 2.6 作废、删除和复制报销单

| 功能 | 请求方式 | 请求地址 | 主要规则 |
| --- | --- | --- | --- |
| 作废 | POST | `/api/reimburse/{id}/void` | 已作废单据不可重复作废 |
| 删除 | DELETE | `/api/reimburse/{id}` | 仅草稿允许逻辑删除 |
| 复制 | POST | `/api/reimburse/{id}/copy` | 生成新的草稿并复制关联行程和分摊 |

## 3. 行程接口

### 3.1 查询行程

- 请求方式：`GET`
- 请求地址：`/api/reimburse/{reimId}/trips`
- 路径参数：`reimId`，报销单主键

### 3.2 新增行程

- 请求方式：`POST`
- 请求地址：`/api/reimburse/{reimId}/trips`
- 请求体：

```json
{
  "travelerId": "EMP_TEST_05",
  "travelerNo": "TEST005",
  "travelerName": "测试人员5",
  "departureCityNo": "10401",
  "departureCityName": "武汉",
  "arrivalCityNo": "10621",
  "arrivalCityName": "上海",
  "arrivalCityType": "1",
  "departureDate": "2026-07-01",
  "arrivalDate": "2026-07-03",
  "tripDescription": "客户现场项目交流"
}
```

业务规则：

- 到达日期不得早于出发日期。
- 新增行程后，系统按日期自动生成补助信息和补助日历。
- 系统重新汇总报销单的补助金额。

### 3.3 修改行程

- 请求方式：`PUT`
- 请求地址：`/api/reimburse/{reimId}/trips/{tripId}`
- 请求体：与新增行程接口相同
- 业务规则：修改后重新生成该行程的补助数据，并重新汇总主表金额。

### 3.4 删除行程

- 请求方式：`DELETE`
- 请求地址：`/api/reimburse/{reimId}/trips/{tripId}`
- 业务规则：删除行程时同步清理关联补助数据，并重新汇总主表金额。

## 4. 补助接口

### 4.1 查询补助及日历

- 请求方式：`GET`
- 请求地址：`/api/reimburse/{reimId}/allowances`
- 返回内容：补助基本信息，以及每一天的餐费、交通和通讯补助日历明细。

### 4.2 调整补助日历

- 请求方式：`PUT`
- 请求地址：`/api/reimburse/{reimId}/allowances/{allowanceId}/calendar`
- 请求体：

```json
[
  {
    "calendarId": "CALENDAR001",
    "mealSelected": 1,
    "mealActualAmount": 80.00,
    "trafficSelected": 0,
    "trafficActualAmount": 0.00,
    "communicationSelected": 1,
    "communicationActualAmount": 30.00
  }
]
```

业务规则：

- 只有草稿状态允许调整。
- 每条系统日历都必须提交，日历 ID 不得重复。
- 未选中的补助项实际金额按 0 处理。
- 选中的实际金额必须大于 0，且不得超过对应标准金额。
- 申请金额按选中项的标准金额汇总，补助金额按实际金额汇总。
- 保存后同步更新主表补助合计和费用分摊金额。

## 5. 费用分摊接口

### 5.1 查询费用分摊

- 请求方式：`GET`
- 请求地址：`/api/reimburse/{reimId}/allocations`

### 5.2 保存费用分摊

- 请求方式：`POST`
- 请求地址：`/api/reimburse/{reimId}/allocations`
- 请求体：

```json
[
  {
    "companyId": "COMP001",
    "companyNo": "GS001",
    "companyName": "初始公司",
    "projectId": "PROJ001",
    "projectNo": "XM001",
    "projectName": "项目A",
    "allocationRatio": 0.6000,
    "allocationAmount": 324.00
  },
  {
    "companyId": "COMP001",
    "companyNo": "GS001",
    "companyName": "初始公司",
    "projectId": "PROJ002",
    "projectNo": "XM002",
    "projectName": "项目B",
    "allocationRatio": 0.4000,
    "allocationAmount": 216.00
  }
]
```

业务规则：

- 分摊比例范围为 `0.0000` 至 `1.0000`。
- 分摊金额不得小于 0。
- 各分摊行比例之和应为 1。
- 各分摊行金额之和应等于报销单总补助金额。
- 页面第一行比例由系统按 `100% - 第2行起比例合计` 自动计算。
- 点击均摊后按行数平均分配，不能整除的比例和金额差值放在首行。
- 页面至少保留一条分摊信息。

### 5.3 删除费用分摊

- 请求方式：`DELETE`
- 请求地址：`/api/reimburse/{reimId}/allocations`
- 说明：删除指定报销单下的全部费用分摊记录。

## 6. 字典和操作日志接口

### 6.1 查询报销业务字典

- 请求方式：`GET`
- 请求地址：`/api/dictionaries/reimbursement`
- 返回内容：员工、部门、费用归属公司、业务类型、项目和城市选项。

### 6.2 查询操作日志

- 请求方式：`GET`
- 请求地址：`/api/reimburse/{reimId}/logs`
- 返回内容：报销单新增、修改、提交、作废和复制等操作记录。

## 7. 常见错误响应

参数校验失败：

```json
{
  "code": 400,
  "message": "报销标题不能为空",
  "data": null
}
```

业务规则校验失败：

```json
{
  "code": 400,
  "message": "只有草稿状态的报销单允许修改",
  "data": null
}
```

系统异常：

```json
{
  "code": 500,
  "message": "系统异常，请稍后重试",
  "data": null
}
```
